# Tune OpenCode review prompts to follow existing Virtru workflow conventions

## Summary
OpenCode reviews are producing false-positive P0 blocking findings that contradict established Virtru conventions:
SHA pinning for internal actions: OpenCode flags virtru-corp/.github/...@v1 as a supply-chain risk requiring SHA pinning, but Virtru policy is to use mutable tags for internally-controlled org actions. Every existing workflow in the repo follows this pattern.
OIDC/WIF trust boundary: OpenCode flags workflows for not constraining OIDC trust at the workflow level, but Virtru enforces this at the GCP IAM attribute condition level, not in individual workflow files.
Permission consistency: OpenCode suggests permissions based on generic best practices that conflict with established patterns in the repo.
Fix: Add Virtru-specific conventions to the review prompt template, CI reviewer agent, security reviewer agent, and soul document so OpenCode checks existing repo patterns before flagging deviations.
